# LCW Spec

> 本文件是 LCW 项目的需求基准，用于迭代时检查是否偏离原始目标。只描述"要什么"和"为什么"，不描述"怎么做"。

## 产品定位

LCW 是一个**基于代码仓库的知识库**，同时服务人和 AI：
- **人**：浏览领域划分、核心业务流程、模型设计、架构全景
- **AI**：回答关于业务和技术的各种问题，并在回答过程中持续修正和增强知识库

核心价值是把散落在几百个 repo 中的隐性知识编织成显性的、可链接的、可查询的结构化知识网络。

## 目标用户

个人使用，未来可能扩展到团队。当前不需要多人协作能力，但知识库内容本身应该可以用 git 共享。

## 规模要求

- 支撑数百到上千个代码仓库
- 不能一次性处理所有 repo，需要增量加载，按构建理解的优先级逐步展开
- 工具应能自动推荐下一批该处理的 repo（基于依赖关系等信号）
- 实际的代码文件不应提交到知识库的 git 仓库

## 能力要求

### 基础能力

| 能力 | 说明 |
|------|------|
| 初始化 | 创建知识库结构 |
| 代码获取 | 克隆和更新代码仓库，管理本地代码副本 |
| 全量摄入 | 从代码中提取结构化知识（架构、模块、接口、领域、流程、词汇） |
| 增量同步 | 根据代码变更增量更新知识库 |
| 健康检查 | 检测知识库与代码的漂移，主动修复 |
| 查询 | 回答问题，同时验证和修正知识库内容 |
| 归档 | 把对话中产生的洞察持久化到知识库 |
| 活动分析 | 按时间切片分析提交历史，支持功能和人员视角 |

### DDD 能力

| 能力 | 说明 |
|------|------|
| 战略层 | 子域划分、限界上下文、上下文映射、统一语言 |
| 战术层 | 聚合根、领域事件、防腐层、领域服务建模 |
| 演进层 | 重构路线图、过渡架构、迁移指南 |
| 漂移检测 | 对比代码现实与 DDD 模型的差异 |

DDD 能力只在用户显式触发时执行，不自动运行。

## 知识结构

### 知识库应该包含什么

知识库中的每类内容都应回答一个清晰的问题：

| 问题 | 内容类型 |
|------|---------|
| 这个系统做什么？ | 业务领域、端到端业务流程、全局架构概览 |
| 怎么构建的？ | 仓库档案、模块详情、跨仓库接口 |
| 用什么词？ | 业务词汇表 |
| 有什么问题？ | 代码事实层发现的问题和技术债 |
| 应该变成什么样？ | DDD 目标模型（战略/战术/演进） |
| 目标和现实差在哪？ | 结构性差异报告 |
| 谁在改什么？ | 提交活动时间线 |

不设置通用的"概念"、"查询结果"、"架构决策"分类——这些内容应融入上述对应类型中，而非独立存在。如果某个洞察无法归入已有分类，说明分类本身可能需要演化。

### 领域页面用中性词汇

领域页面属于实然层，描述的是代码实际呈现的结构。模板应使用中性词汇（"核心数据模型"、"对外通知/消息"、"与其他领域的依赖"），避免强制套用 DDD 专有概念（"聚合根"、"领域事件"、"命令"）。DDD 视角的分析放在 ddd/ 目录或领域页面底部的可选 DDD 章节中。

### 全局概览是入口不是汇总

全局概览页只保留高层全景（业务能力地图、领域关系、系统全景），不堆砌详细的仓库职责表、接口汇总、依赖图——这些信息已经在各仓库页面和接口页面中维护。概览是导航起点，读者通过链接深入。

### 模块按仓库分组

模块页面按仓库组织子目录，避免上千个文件平铺在一个目录中。

### issues 和 gaps 的边界

两者都记录"问题"，但视角不同：
- **issues**：代码事实层发现的问题（实然）——技术债、接口矛盾、覆盖缺失。由 ingest/sync/lint 发现。
- **gaps**：目标模型与代码现实的结构性差异（应然 vs 实然）——上下文边界不一致、模块错分、非法耦合。由 DDD 分析发现。

一个是"代码本身有问题"，一个是"代码跟目标模型不一致"。

### 人类输入需要单独管理

用户在对话中提供的、代码中不存在的信息（业务背景、历史决策原因、团队约定等）是一种特殊输入。这类信息可能不准确、会过时，不应直接混入代码事实页面。

知识库需要一个专门的位置存放人类输入，带时间戳和来源标注。这些内容作为"待验证的线索"，在后续 query/ingest/sync 过程中如果能用代码验证才融入对应的 wiki 页面。

## 核心原则

### 实然/应然分离

这是最重要的原则。

- **实然（As-Is）**：从代码中提取的事实——模块结构、接口关系、数据模型、业务流程。反映"代码现在是什么样"。
- **应然（To-Be）**：DDD 分析产出的目标模型——子域划分、限界上下文、聚合根设计。反映"领域模型应该是什么样"。

这个边界必须在写入和回答中同时维持：
- 写入时不混：提取代码事实的操作不修改 DDD 产出，DDD 操作不篡改代码事实
- 回答时标注：引用代码事实直接陈述，引用 DDD 产出明确标注来源
- 冲突时呈现：两者不一致时向用户展示差异，不替用户做判断
- 绝不把目标设计当成代码现状来描述

### 查询即维护

查询不是只读操作。查询时自动校验代码、确认知识库内容与代码一致，不一致时自动修复知识库。具体链路：

1. 用户提问 → 自动触发查询（无需手动指定命令）
2. 查询过程中回溯源码，验证知识库描述是否仍然正确
3. 发现不一致 → 自动修正知识库页面
4. 修正知识库时写 log（所有对知识库的修改都必须写 log）
5. 回答用户问题

知识库是活的，越用越准。

### 所有修改必须写 log

任何对知识库的写入操作——无论来自 ingest、sync、lint、query 的自动修正、file 的归档、还是 ddd 的分析产出——都必须在 log.md 中记录。log 是人类了解知识库演化时间线的唯一入口。

### 知识库是升维产物

500 行代码 → 20 行知识。提取架构理解，不复制实现细节。引用源码用路径，不粘贴代码。关系网络比单个页面更重要。

## 触发模式

- **查询**：默认行为，自动触发。查询过程中自动校验代码并修复知识库（用户无需显式指定命令）
- **其他所有操作**（初始化、拉取、摄入、同步、检查、归档、活动分析、DDD）：必须手动触发，不自动执行

## 分发与安装

### 目标

- **项目级**，不是全局：LCW 只在显式启用的项目中可用，不污染其他项目的上下文
- **多宿主**：支持 Claude Code、Codex、OpenCode、Windsurf 等宿主工具，不依赖任何工具特有机制（如 `${CLAUDE_SKILL_DIR}`）
- **自包含**：技能目录（SKILL.md + 参考文件 + 模板 + agent）是一个完整单元，可以被安装到任意项目中正常工作，不依赖外部绝对路径
- **易于维护和更新**：用户能用标准工具安装和更新，不需要手动复制文件

为什么不做全局：LCW 设计用于跨多代码库整合看一大片领域，是特定工作空间的工具。全局注册会在不相关的项目中引入无意义的触发候选。

### 安装方式

只支持项目级安装，不提供全局安装。

安装通过 `setup` 脚本完成：在目标项目目录执行 setup，创建从项目的宿主工具 skills 目录到 LCW 技能目录的 symlink。setup 脚本通过自身路径自动定位 LCW 仓库，不假设安装位置（推荐的 `~/.lcw` 路径在 README 中说明，不硬编码在工具里）。

- 不指定宿主工具时，默认安装到 Claude Code（`.claude/skills/lcw`）
- 通过 `--host` 指定其他宿主工具（codex、opencode、windsurf），可多次指定
- 更新方式：在 LCW 仓库中 `git pull`，symlink 自动生效

## 不做的事

- 不做代码搜索（有专门的工具）
- 不做代码修改（代码副本只读）
- 不做实时监控（知识库是快照，通过同步操作手动刷新）
- 不在知识库中粘贴源代码
- 不自动执行重操作（摄入、同步、DDD 等必须用户确认）
