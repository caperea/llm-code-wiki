# LCW Spec

> 本文件是 LCW 项目的需求基准，用于迭代时检查是否偏离原始目标。只描述"要什么"和"为什么"，不描述"怎么做"。

## 产品定位

LCW 是一个**基于代码仓库的知识库**，同时服务人和 AI：
- **人**：浏览领域划分、核心业务流程、模型设计、架构全景
- **AI**：回答关于业务和技术的各种问题，并在回答过程中持续修正和增强知识库

核心价值是把散落在几百个 repo 中的隐性知识编织成显性的、可链接的、可查询的结构化知识网络。

## 目标用户

个人使用，未来可能扩展到团队。当前不需要多人协作能力，但知识库内容本身应该可以用 git 共享。

## 规模要求

- 支撑数百到上千个代码仓库
- 不能一次性处理所有 repo，需要增量加载，按构建理解的优先级逐步展开
- 工具应能自动推荐下一批该处理的 repo（基于依赖关系等信号）
- 实际的代码文件不应提交到知识库的 git 仓库

## 能力要求

### 基础能力

| 能力 | 说明 |
|------|------|
| 初始化 | 创建知识库结构 |
| 代码获取 | 克隆和更新代码仓库，管理本地代码副本 |
| 全量摄入 | 从代码中提取结构化知识（架构、模块、接口、领域、流程、词汇） |
| 增量同步 | 根据代码变更增量更新知识库 |
| 健康检查 | 检测知识库与代码的漂移，主动修复 |
| 查询 | 回答问题，同时验证和修正知识库内容 |
| 归档 | 把对话中产生的洞察持久化到知识库 |
| 活动分析 | 按时间切片分析提交历史，支持功能和人员视角 |

### DDD 能力

| 能力 | 说明 |
|------|------|
| 战略层 | 子域划分、限界上下文、上下文映射、统一语言 |
| 战术层 | 聚合根、领域事件、防腐层、领域服务建模 |
| 演进层 | 重构路线图、过渡架构、迁移指南 |
| 漂移检测 | 对比代码现实与 DDD 模型的差异 |

DDD 能力只在用户显式触发时执行，不自动运行。

## 核心原则

### 实然/应然分离

这是最重要的原则。

- **实然（As-Is）**：从代码中提取的事实——模块结构、接口关系、数据模型、业务流程。反映"代码现在是什么样"。
- **应然（To-Be）**：DDD 分析产出的目标模型——子域划分、限界上下文、聚合根设计。反映"领域模型应该是什么样"。

这个边界必须在写入和回答中同时维持：
- 写入时不混：提取代码事实的操作不修改 DDD 产出，DDD 操作不篡改代码事实
- 回答时标注：引用代码事实直接陈述，引用 DDD 产出明确标注来源
- 冲突时呈现：两者不一致时向用户展示差异，不替用户做判断
- 绝不把目标设计当成代码现状来描述

### 查询即维护

查询不是只读操作。每次查询都应回溯源码验证知识库描述是否仍然正确，发现不一致时立即修正。知识库是活的，越用越准。

### 知识库是升维产物

500 行代码 → 20 行知识。提取架构理解，不复制实现细节。引用源码用路径，不粘贴代码。关系网络比单个页面更重要。

## 触发模式

- **查询**：默认行为，应能自动触发（用户提问时无需显式指定命令）
- **其他所有操作**（初始化、拉取、摄入、同步、检查、归档、活动分析、DDD）：必须手动触发，不自动执行

## 不做的事

- 不做代码搜索（有专门的工具）
- 不做代码修改（代码副本只读）
- 不做实时监控（知识库是快照，通过同步操作手动刷新）
- 不在知识库中粘贴源代码
- 不自动执行重操作（摄入、同步、DDD 等必须用户确认）
