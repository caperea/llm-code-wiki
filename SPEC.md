# LCW Spec

> 本文件是 LCW 项目的需求基准，用于迭代时检查是否偏离原始目标。只描述"要什么"和"为什么"，不描述"怎么做"。

## 产品定位

LCW 是一个**基于代码仓库的知识库**，同时服务人和 AI：
- **人**：浏览领域划分、核心业务流程、模型设计、架构全景
- **AI**：回答关于业务和技术的各种问题，并在回答过程中持续修正和增强知识库

核心价值是把散落在几百个 repo 中的隐性知识编织成显性的、可链接的、可查询的结构化知识网络。

底层思想来自 Andrej Karpathy 的 LLM Wiki 概念（见 `llm-wiki.md`），但那只是一个引子和初步想法。本项目在此基础上大幅演化，SPEC 是当前的需求基准，与 llm-wiki.md 不一致时以 SPEC 为准。

## 目标用户

个人使用，未来可能扩展到团队。当前不需要多人协作能力，但知识库内容本身应该可以用 git 共享。

## 浏览方式

Wiki 的人类浏览端是 Obsidian。wiki 项目目录直接作为 Obsidian vault 打开，人通过 Obsidian 浏览页面、跟踪链接、查看 graph view。这意味着：

- 所有 wiki 页面使用 Obsidian 兼容的 markdown 格式和 `[[wikilink]]` 语法
- 页面应对人类阅读友好（不是只给 LLM 看的中间产物）
- Obsidian 插件（graph view、Dataview、Marp 等）是合法的消费方式，页面的 frontmatter 和结构应便于这些工具使用

## 规模要求

- 支撑数百到上千个代码仓库
- 不能一次性处理所有 repo，需要增量加载，按构建理解的优先级逐步展开
- 工具应能自动推荐下一批该处理的 repo（基于依赖关系等信号）
- 实际的代码文件不应提交到知识库的 git 仓库
- **已知限制**：当前 repo 以名称做唯一标识（`.sources/{name}/`、`repos/{name}.md`、`modules/{name}/`）。不同来源的同名 repo（如 `team-a/common` 和 `team-b/common`）会冲突。未来需要支持别名或 `{org}/{repo}` 命名策略

## 能力要求

### 基础能力

| 能力 | 说明 |
|------|------|
| 初始化 | 创建知识库结构 |
| 代码获取 | 克隆和更新代码仓库，管理本地代码副本 |
| 全量摄入 | 从代码中提取结构化知识（架构、模块、接口、领域、流程、词汇） |
| 增量同步 | 根据代码变更增量更新知识库 |
| 健康检查 | 检测知识库与代码的漂移，主动修复 |
| 查询 | 回答问题，同时验证和修正知识库内容 |
| 归档 | 把对话中产生的洞察持久化到知识库 |
| 执行计划 | 跨命令的全局视图：一次看到所有 repo 该 pull/ingest/sync/lint/skip。无参数时输出全量矩阵，指定命令名时只看该命令的待办（如 `plan ingest`）。主要用于初始构建或大批量新增 repo 时 |
| 结构迁移 | 将旧版 wiki 迁移到当前目录结构和 schema，不假设旧结构，现场分析比对，计划先行，git 快照后执行 |
| 合并 | 将多个独立的 wiki 项目平等合并成一个新的统一知识库（无主从）。前置要求 schema 版本一致。检测 repo 重叠、领域边界冲突、词汇表差异，发现跨 wiki 接口。源 wiki 只读不修改。全程通过 `merge-plan.md` 维护目标、决策、逐项进度和备注，支持中断恢复和独立校验。词汇表采用"机械合并 + 标记冲突"策略（同名条目合并变体取并集，定义不同则标记"不一致"），不替用户做统一判断——由后续 lint 和 query 回溯代码逐步消解 |

### DDD 能力

| 能力 | 说明 |
|------|------|
| 战略层 | 子域划分、限界上下文、上下文映射、统一语言 |
| 战术层 | 聚合根、领域事件、防腐层、领域服务建模 |
| 演进层 | 重构路线图、过渡架构、迁移指南 |
| 漂移检测 | 对比代码现实与 DDD 模型的差异 |

DDD 能力只在用户显式触发时执行，不自动运行。

## 知识结构

### 知识库应该包含什么

知识库中的每类内容都应回答一个清晰的问题：

| 问题 | 内容类型 |
|------|---------|
| 这个系统做什么？ | 业务领域、端到端业务流程、全局架构概览 |
| 系统之间怎么协作？ | 系统拓扑（多个 repo 组成的系统单元、系统间关系） |
| 怎么构建的？ | 仓库档案、模块详情、跨仓库接口 |
| 用什么词？ | 业务词汇表 |
| 有什么问题？ | 代码事实层发现的问题和技术债 |
| 应该变成什么样？ | DDD 目标模型（战略/战术/演进） |
| 目标和现实差在哪？ | 结构性差异报告 |

不设置通用的"概念"、"查询结果"、"架构决策"分类——这些内容应融入上述对应类型中，而非独立存在。如果某个洞察无法归入已有分类，说明分类本身可能需要演化。

### 领域页面用中性词汇

领域页面属于实然层，描述的是代码实际呈现的结构。模板应使用中性词汇（"核心数据模型"、"对外通知/消息"、"与其他领域的依赖"），避免强制套用 DDD 专有概念（"聚合根"、"领域事件"、"命令"）。DDD 视角的分析放在 ddd/ 目录或领域页面底部的可选 DDD 章节中。

### 全局概览是入口不是汇总

全局概览页只保留高层全景（系统全景、业务能力地图、参与者与角色、领域关系、核心业务流程），不堆砌详细的仓库职责表、接口汇总、依赖图——这些信息已经在各仓库页面和接口页面中维护。概览是导航起点，读者通过链接深入。每一行信息应该是索引性的（帮你找到去哪看），而不是内容性的（把别处的内容搬过来）。

### 模块按仓库分组

模块页面按仓库组织子目录，避免上千个文件平铺在一个目录中。

### issues 和 gaps 的边界

两者都记录"问题"，但视角不同：
- **issues**：代码事实层发现的问题（实然）——技术债、接口矛盾、覆盖缺失。由 ingest/sync/lint 发现。
- **gaps**：目标模型与代码现实的结构性差异（应然 vs 实然）——上下文边界不一致、模块错分、非法耦合。由 DDD 分析发现。

一个是"代码本身有问题"，一个是"代码跟目标模型不一致"。

### 人类输入属于 sources 层

用户在对话中提供的信息（业务背景、历史决策原因、团队约定等）和查询记录（提问角度、思考路径）都是 sources，不是 wiki。它们存放在 `.inputs/` 中（见"Sources 的组成"），带时间戳和来源标注。

这些内容作为构建和校验 wiki 的原料：在 ingest/sync/query 过程中如果能用代码验证，将信息融入 wiki 页面（可追溯到 source）；无法验证的保留在 `.inputs/` 中，不进入 wiki。

### legacy 目录的定位

`legacy/` 仅在执行过结构迁移时才存在，存放旧结构中无法自动归类到当前分类体系的页面。这些内容保留原样供参考，但不再由任何命令维护——不更新、不同步、不检查一致性。查询时如果引用了 legacy/ 中的内容，应标注其来源和低置信度。

## 核心原则

### 实然/应然分离

这是最重要的原则。

- **实然（As-Is）**：从代码中提取的事实——模块结构、接口关系、数据模型、业务流程。反映"代码现在是什么样"。
- **应然（To-Be）**：DDD 分析产出的目标模型——子域划分、限界上下文、聚合根设计。反映"领域模型应该是什么样"。

这个边界必须在写入和回答中同时维持：
- 写入时不混：提取代码事实的操作不修改 DDD 产出，DDD 操作不篡改代码事实
- 回答时标注：引用代码事实直接陈述，引用 DDD 产出明确标注来源
- 冲突时呈现：两者不一致时向用户展示差异，不替用户做判断
- 绝不把目标设计当成代码现状来描述

### 查询即维护

查询不是只读操作。查询时自动校验代码、确认知识库内容与代码一致，不一致时自动修复知识库。具体链路：

1. 用户提问 → 自动触发查询（无需手动指定命令）
2. 查询过程中回溯源码，验证知识库描述是否仍然正确
3. 发现不一致 → 自动修正知识库页面
4. 修正知识库时写 log（所有对知识库的修改都必须写 log）
5. 回答用户问题

知识库是活的，越用越准。

### 越用越聪明

每次对话都让知识库变得更好。不仅在查询过程中自动修正过时内容（见"查询即维护"），对话结束时还自动执行三项检查：

1. **新认知 → wiki**：对话中产生了 wiki 中没有的新理解？更新相关页面
2. **有价值的 query → .inputs/queries/**：这个问题的提问角度能帮助未来的 ingest/sync 主动构建知识？保存问题和发现路径
3. **用户反馈 → .inputs/notes/**：用户提供了代码中找不到的信息（业务背景、历史原因、团队约定）？保存为 source

这三项是静默执行的——不需要用户触发 `/lcw file`，不需要用户意识到这个过程。`/lcw file` 仍然存在，用于用户主动归档更复杂的洞察。

`.inputs/` 不是死档案。ingest/sync 在分析代码时会读取 `.inputs/queries/` 和 `.inputs/notes/`，用过去的提问和反馈指导当前分析——优先覆盖被问过的领域，利用用户提供的业务背景解读代码。

### 自愈

wiki 中的所有问题——过时描述、断链、重复条目、不一致标记、冲突残留——都不依赖人工清理。lint 和 query 在日常运行中自动发现并修复，判断依据是代码，不是猜测。

具体来说：

- **页面内容过时**：query 回溯源码发现 wiki 描述与代码不符 → 立即修正
- **词汇表重复/不一致**：回溯代码理解实际含义。含义相同 → 合并；含义不同 → 标记"多义"。"不一致"状态的条目如果代码已趋于统一 → 自动消解
- **断链和孤立页面**：lint 扫描 wikilink → 自动修复路径或补充链接
- **领域边界漂移**：代码变更导致模块归属变化 → lint 修正 domains/ 页面
- **接口变更**：上下游关系变化 → lint/query 更新 interfaces/ 页面
- **合并残留**（merge 后）：冲突标记、unverified 接口 → lint 回溯代码逐步消解
- **issues 过时**：问题在代码中已修复 → lint 自动标记 resolved

query 遇到问题时随手修，lint 在健康检查时批量扫。两者合力保证 wiki 越用越干净。不能用代码判断的问题（如领域划分合理性）标记出来等人决策，不勉强自动处理。

### 所有修改必须写 log

任何对知识库的写入操作——无论来自 ingest、sync、lint、query 的自动修正、file 的归档、还是 ddd 的分析产出——都必须在 log.md 中记录。log 是人类了解知识库演化时间线的唯一入口。

### 知识库是升维产物

500 行代码 → 20 行知识。提取架构理解，不复制实现细节。引用源码用路径，不粘贴代码。关系网络比单个页面更重要。

### Wiki 是派生物，Sources 是基底

Wiki 的所有内容都应该能从 sources 重建。如果把 wiki 全部删掉，从 sources 重新跑 ingest 应该能恢复出等价的知识库。这意味着：

- **查询产生的洞察**：如果能融入 wiki 已有分类（domains/、modules/、flows/ 等），直接更新进去。查询本身（提问角度、思考路径）作为 source 保存——它是构建知识的原料，不是知识本身。
- **用户提供的信息**（业务背景、历史决策、团队约定等）：是 source，不是 wiki。独立维护，用于构建和校验 wiki，但不直接出现在 wiki 页面中。
- **Wiki 页面不存储不可重建的内容**：任何写入 wiki 的信息，其来源（代码或 source）必须可追溯。

### Sources 的组成

Sources 是 wiki 的输入层，包含三类：

| 类型 | 说明 | 存储 |
|------|------|------|
| 源代码仓库 | `.sources/` 中的克隆代码，每个 repo 有自己的 git | gitignored，不提交到 wiki 仓库 |
| 用户提问 | 查询中产生的问题、思考角度、分析路径 | 提交到 wiki 仓库的 git |
| 用户输入 | 人提供的代码中不存在的信息（业务背景、决策原因等） | 提交到 wiki 仓库的 git |

Sources 不在查询时直接展示，但用于信息补充、校验、和重建 wiki。

## 触发模式

- **查询**：默认行为，自动触发。查询过程中自动校验代码并修复知识库（用户无需显式指定命令）
- **其他所有操作**（初始化、拉取、摄入、同步、检查、归档、DDD、迁移）：必须手动触发，不自动执行

## 分发与安装

### 目标

- **项目级**，不是全局：LCW 只在显式启用的项目中可用，不污染其他项目的上下文
- **多宿主**：支持 Claude Code、Codex、OpenCode、Windsurf 等宿主工具，不依赖任何工具特有机制（如 `${CLAUDE_SKILL_DIR}`）
- **自包含**：技能目录（SKILL.md + 参考文件 + 模板 + agent）是一个完整单元，可以被安装到任意项目中正常工作，不依赖外部绝对路径
- **易于维护和更新**：用户能用标准工具安装和更新，不需要手动复制文件

为什么不做全局：LCW 设计用于跨多代码库整合看一大片领域，是特定工作空间的工具。全局注册会在不相关的项目中引入无意义的触发候选。

### 安装方式

只支持项目级安装，不提供全局安装。

安装通过 `setup` 脚本完成：在目标项目目录执行 setup，创建从项目的宿主工具 skills 目录到 LCW 技能目录的 symlink。setup 脚本通过自身路径自动定位 LCW 仓库，不假设安装位置（推荐的 `~/.lcw` 路径在 README 中说明，不硬编码在工具里）。

- 不指定宿主工具时，默认安装到 Claude Code（`.claude/skills/lcw`）
- 通过 `--host` 指定其他宿主工具（codex、opencode、windsurf），可多次指定
- 更新方式：在 LCW 仓库中 `git pull`，symlink 自动生效

## 不做的事

- 不做代码搜索（有专门的工具）
- 不做代码修改（代码副本只读）
- 不做实时监控（知识库是快照，通过同步操作手动刷新）
- 不在知识库中粘贴源代码
- 不自动执行重操作（摄入、同步、DDD 等必须用户确认）
