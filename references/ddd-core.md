# 反向 DDD 梳理——共享基础

> 本文件是所有 `/lcw ddd` 子命令的共享基础。每次 DDD 操作前先读本文件，再按子命令读对应的层级文件。

## 核心理念

你做的是**反向 DDD**——从代码考古出发重建领域模型，不是从需求出发设计领域模型。

最大的陷阱是**被代码现状带跑**：把烟囱当领域，把实现细节当业务概念。对抗这个陷阱的方法是**双向校验**：

- **自下而上**：从 `__wiki__/` 中提取代码事实（模块、接口、数据结构、调用关系）
- **自上而下**：从业务能力出发推导领域应该是什么样的
- **差异分析**：两边对不上的地方才是真正需要关注的问题

wiki 中的内容遵循**三层视角模型**：

- **现状理解层**（`__wiki__/` 根目录）：代码中实际呈现的结构。由 ingest/diff/lint 维护。
- **目标设计层**（`__wiki__/ddd/`）：领域模型应该是什么样。由 ddd 命令维护。
- **差距层**（`glossary.md` 的"差距"字段 + `ddd/gaps.md`）：两者的不一致 = 治理清单。

特殊的**双视角文档**：`glossary.md` 同时包含现状用法（As-Is）和领域定义（To-Be）。统一语言不单独建文件，直接增强 glossary.md 的条目。

## 命令结构

```
/lcw ddd                          # 交互式全流程
/lcw ddd strategic                # 战略层：领域识别
/lcw ddd tactical <context-name>  # 战术层：指定上下文的模型落地
/lcw ddd evolution                # 演进层：重构推进规划
/lcw ddd audit                    # 漂移检测：代码现实 vs DDD 模型
```

无参数时的默认行为：
1. 检查 `__wiki__/ddd/` 是否已有产出
   - 有 → 读取 `ddd/status.md` 进度，从上次断点继续
   - 无 → 从先决判断开始
2. 战略层完成后，交互式询问：先做哪个上下文的战术层？
3. 战术层完成一个后提示：继续下一个还是进入演进规划？

**迭代节奏**：战略层粗做一遍 → 挑一个上下文做战术层 → 反过来修正战略层 → 再做下一个上下文。这个往返是正常的，不要等战略层"完美"再开始战术层。

**统一语言是贯穿活动**：不是某个步骤做完才开始收集术语。从步骤 1 业务能力盘点起就持续增量更新 `glossary.md`——每个分析步骤结束时都检查是否有新术语、新歧义、新定义需要记录。统一语言的质量直接决定后续所有分析的准确度。

## 状态总览

每次 DDD 操作（strategic / tactical / evolution / audit）完成后，更新 `ddd/status.md`。这是 DDD 分析的仪表盘——新读者打开这一个文件就能了解整体进度，不用逐个翻阅产出文件。

页面内容：

- **梳理范围**：纳入的 repo 清单（链接到 `ddd/decisions.md`）
- **进度表**：

```markdown
| 层次 | 产出 | 状态 | 最后更新 |
|------|------|------|---------|
| 先决判断 | [[ddd/decisions]] | ✓ 已完成 | 2026-05-01 |
| 战略-全景图 | [[ddd/panorama]] | ✓ 已确认 | 2026-05-02 |
| 战略-上下文 | [[ddd/contexts]] | ✓ 已确认 | 2026-05-03 |
| 战略-差异 | [[ddd/gaps]] | ✓ 已完成 | 2026-05-03 |
| 战术-OrderContext | [[ddd/tactical/order]] | 进行中 | 2026-05-08 |
| 战术-PaymentContext | — | 未开始 | — |
| 演进-路线图 | — | 未开始 | — |
| 最近审计 | [[ddd/audit-2026-05-10]] | 2026-05-10 | — |
```

- **关键发现摘要**：3-5 条最重要的发现（链接到详细页面）
- **下一步行动**：当前应该做什么

### ddd/status.md 模板

```yaml
---
type: ddd-status
created: {YYYY-MM-DD}
last_updated: {YYYY-MM-DD}
---
```

必含章节：梳理范围、进度表、关键发现摘要、下一步行动。

## 与 wiki 的集成约定

### 读取（DDD → wiki）

| wiki 页面 | DDD 怎么用 |
|-----------|-----------|
| `domains/*.md` | 子域划分的原料，但 DDD 可能重新划分边界 |
| `interfaces/*.md` | 复用 relationship 标注，补充遗漏的映射关系 |
| `glossary.md` | 统一语言的起点 |
| `modules/*.md` 的 `model_style` | 贫血检测的直接输入 |
| `flows/*.md` | 领域事件提取的主要来源（Event Storming 结构） |
| `activities/*.md` | 变更频率、团队边界的数据源 |
| `issues/*.md` | 问题密度评估 |

### 回写（DDD → wiki）

DDD 分析过程中发现的内容回写到 wiki：

- 新发现的问题 → 创建 `issues/*.md`
- 术语的领域定义和差距 → 更新 `glossary.md` 条目的 DDD 字段（领域定义、差距、决策、消解状态）
- 新发现的概念 → 创建 `concepts/*.md`
- 领域的 DDD 定位 → 更新 `domains/*.md` 的 `ddd_context`/`ddd_status` frontmatter 字段 + 底部"DDD 视角"章节

DDD 不修改 `modules/*.md`、`repos/*.md` 的主体内容——这些是代码事实，由 ingest/diff/lint 维护。对于 `domains/*.md`，DDD 只写入 frontmatter 的 DDD 字段和底部的"DDD 视角"章节，不修改主体内容。对于 `glossary.md`，DDD 只写入 DDD 字段，不修改"现状用法"。

**例外**：DDD 分析确认了上下文映射关系后，可以回写 `interfaces/*.md` 的 `relationship`、`upstream`、`downstream` 字段——这属于"代码事实的补充"（从代码中可观测但 ingest 未捕获的关系类型），而非 DDD 目标态。

### 日志

所有 DDD 操作在 `log.md` 中记录：

```

- 烟囱: K 组
- 产出: ddd/panorama.md, ddd/contexts.md, ddd/gaps.md
- 更新: glossary.md（补充 DDD 字段 N 条）, domains/*.md（补充 DDD 视角 M 个）

## [ISO时间] ddd tactical | {context-name}
- 聚合根: N 个
- 领域事件: M 个
- 防腐层: K 个
- 产出: ddd/tactical/{context-name}.md

## [ISO时间] ddd evolution | 演进规划
- 评估上下文: N 个
- 优先重构: M 个
- 产出: ddd/evolution/roadmap.md, ...

## [ISO时间] ddd audit | 漂移检测
- 检查项: N 个
- 差异: M 个（正面: A, 偏离: B, 模型过时: C）
- 产出: ddd/audit-{YYYY-MM-DD}.md
```

## 常见陷阱

在执行过程中时刻警惕：

1. **把烟囱当领域**：三个团队各自做了一套"风控"，不代表有三个风控域——很可能只有一个核心风控域和两个应该被合并的烟囱
2. **把贫血模型当领域模型**：wiki 抽出来的数据结构如果直接拿来当领域模型，大概率是贫血的。必须把"行为"补回模型里
3. **回避难听的结论**：大量的"Separate Ways"和"ACL"在老系统里是常态，不要美化为"Partnership"
4. **子域分类一刀切**：团队容易把自己负责的所有东西都标为"核心域"。核心域的判据是"差异化价值"，不是"重要性"
5. **忽略团队边界**：康威定律是真实的。上下文划分必须考虑团队结构，否则会产生"组织上无法维护"的领域模型
6. **过早追求完美**：战略层的上下文划分永远不会在第一遍就完美。先做粗分，在战术层验证后再修正
7. **产出文档化但不被使用**：DDD 的最终产出不是文档，是团队共识。通过 audit 机制保持 DDD 模型与代码的持续对齐，让文档保持活性
8. **统一语言放到最后才做**：术语不一致往往是领域边界问题的最早信号。如果把统一语言当成最后一步，前面的子域分类和上下文划分可能已经建立在模糊的术语基础上了
