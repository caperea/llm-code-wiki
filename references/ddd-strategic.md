# 反向 DDD——战略层

> 前置依赖：先读 `references/ddd-core.md`。本文件覆盖先决判断 + 战略层分析步骤 + 相关页面模板。

## 第零步：先决判断

首次运行 `/lcw ddd` 时，交互式引导用户做两个决策，持久化到 `ddd/decisions.md`。已有决策文件时跳过此步（但用户可随时用 `/lcw ddd` 重新审视）。

### 1. 梳理范围与目标

- **范围**：哪些 repo 纳入本次梳理？从 `__wiki__/repos/` 中列出清单供用户选择
- **目标**：梳理要解决什么问题？
  - 理解现状（新人 onboarding / 团队对齐）
  - 规划重构（烟囱合并 / 架构升级）
  - 支撑新需求设计（在已有架构上长出新能力）
  - 持续治理（建立领域模型并保持与代码同步）
- **输出偏好**：词汇表语言（中/英/双语）、图表格式（ASCII art / Mermaid）

### 2. 系统架构定位（仅横切系统需要）

仅当系统属于横切类型（治理、安全、合规、监控等嵌入所有业务的系统）时，需要选择架构模式：集中式平台 / 分布式框架 / 两层分离。纯业务系统标"不适用"跳过。这个判断不先做，后面的限界上下文划分会反复推翻。

## 第一层：战略层

命令：`/lcw ddd strategic`

### 输入

从 `__wiki__/` 中读取：

| 页面类型 | 用途 |
|---------|------|
| `overview.md` | 系统全景、业务能力地图 |
| `domains/*.md` | 已识别的领域（代码事实层面） |
| `interfaces/*.md` | 跨系统接口、已有 DDD 关系类型标注 |
| `glossary.md` | 术语表 |
| `flows/*.md` | 端到端业务流程 |
| `modules/*.md` | 模块详情（按需深入） |
| `issues/*.md` | 已知问题（可能揭示领域边界问题） |

如果 wiki 尚未摄入或内容不足，提示用户先执行 `/lcw ingest`。

### 分析步骤

#### 1. 业务能力盘点

从 wiki 的 domains 和 modules 中提取系统对外暴露的所有业务能力：

- 读取每个 `domains/*.md` 的 `capabilities` 字段
- 读取每个 `modules/*.md` 的 Public API 章节
- 去重合并，按业务含义（而非代码结构）分组
- 标注每个能力：对外暴露 / 仅内部使用 / 未明确

产出：能力清单（写入 `ddd/panorama.md` 的"业务能力盘点"章节）

#### 2. 子域分类

将能力分组为三类：

| 类型 | 判据 | 投入策略 |
|------|------|---------|
| 核心域 | 直接产生业务差异化价值的能力 | 重点投入、精心建模 |
| 支撑域 | 业务必需但不直接产生差异化的能力 | 标准化、够用即可 |
| 通用域 | 通用基础设施能力（无业务特殊性） | 尽量复用外部方案 |

分类时警惕两个常见错误：
- 把所有东西都标为"核心"——如果一切都是核心，就没有核心
- 把重要性等同于核心域——"重要"不等于"差异化"，数据库很重要但通常是通用域

这一步需要人工确认——呈现分类方案，说明判据，让用户调整。

产出：更新 `ddd/panorama.md` 的"子域分类"章节

#### 3. 烟囱检测

在子域分类的基础上，找"实质上做同一件事"的多个实现：

- 在 wiki 中搜索相似的 API 签名、数据结构、业务逻辑描述
- 对比不同模块中名称相似或职责重叠的组件
- 重点关注：同一业务操作在不同入口（HTTP API / RPC / 定时任务 / 消息消费者）的多份实现
- 对每个烟囱记录：存在于哪些模块、差异点、统一的阻碍

烟囱检测放在子域分类之后，因为需要先有领域上下文才能区分"跨域的合理重复"和"同域的无意烟囱"。

产出：写入 `ddd/panorama.md` 的"烟囱报告"章节

> **Checkpoint 1：子域 + 烟囱确认**
>
> 到这里暂停，向用户呈现：
> 1. 业务能力清单（按子域分组）
> 2. 核心域 / 支撑域 / 通用域的分类方案及判据
> 3. 烟囱检测的发现
>
> 等用户确认或调整后再继续。子域分类是后续所有工作的基础——如果这里分错了，限界上下文、上下文映射、统一语言全部会歪。调整成本在这里最低，往后越来越高。

#### 4. 限界上下文划分

基于能力分组 + 数据边界 + 团队边界 + 烟囱分析，提出上下文划分方案：

- 一个限界上下文 = 一组内聚的业务能力 + 统一的领域语言 + 明确的边界
- 边界判据：
  - 数据边界：哪些实体只在这个上下文内被修改？
  - 语言边界：同一个词在不同地方含义不同？（从 glossary.md 的"多义"条目获取）
  - 团队边界：谁负责这部分代码？（从 activities/ 的作者数据获取）
  - 变更频率：一起变更的代码通常属于同一个上下文

对每个上下文写清楚：
- 职责边界（做什么、不做什么）
- 对外契约（暴露什么接口）
- 拥有的核心数据（哪些实体归这个上下文"拥有"）

#### 5. 上下文映射

从 `interfaces/*.md` 中提取现有的跨上下文关系，补充遗漏的：

| 关系类型 | 含义 | 老系统常见度 |
|---------|------|------------|
| Partnership | 两个上下文紧密合作，同步演进 | 低 |
| Customer-Supplier | 上游提供，下游消费，上游主导 | 中 |
| Conformist | 下游完全迁就上游模型 | 中 |
| Anti-Corruption Layer | 下游用转换层隔离上游的脏模型 | 高 |
| Shared Kernel | 两个上下文共享一小部分模型 | 中 |
| Open Host Service | 上游提供标准化协议供多下游使用 | 中 |
| Separate Ways | 各自独立，不集成 | 高 |

老系统改造里"Separate Ways"和"ACL"用得最多——不要回避承认这一点。

步骤 4 和 5 的产出合并写入 `ddd/contexts.md`（上下文定义 + 映射关系在同一文件中，因为它们天然一起阅读和维护）。

> **Checkpoint 2：限界上下文确认**
>
> 到这里再次暂停，向用户呈现：
> 1. 限界上下文清单（每个上下文的职责边界、对外契约、核心数据）
> 2. 上下文映射全景图（关系类型 + 集成方式）
> 3. 与 wiki `domains/` 现有划分的对比——哪些一致、哪些要拆分或合并
> 4. 边界判据说明（为什么这样划分而非那样划分）
>
> 等用户确认后再做统一语言整合和差异报告。上下文边界一旦确定，后续的统一语言、战术层建模都基于此。如果用户对边界有异议，现在改成本最低。

#### 6. 统一语言整合

> 注意：统一语言是贯穿整个战略层的持续活动（见 ddd-core.md "迭代节奏"）。这一步不是"开始做统一语言"，而是在前 5 步持续收集的基础上做一次系统性整合。

直接增强 `glossary.md`（双视角文档），不单独生成词汇表文件。

以 `glossary.md` 现有条目为基础：

- 提取所有 status 为"不一致"和"多义"的术语
- 对每个核心名词，在条目中补充 DDD 字段：
  - **领域定义（To-Be）**：该术语在领域模型中的精确定义。如果需要拆分为多个概念，列出每个概念的名称和定义
  - **差距**：现状用法与领域定义之间的不一致
  - **决策**：消解差距的行动方向
  - **消解状态**：未开始 / 进行中 / 已消解
- 标注哪些不一致是"历史遗留但无害"，哪些是"必须修正否则会持续引发误解"

写入时**保留** glossary.md 中由 ingest/diff/lint 维护的"现状用法"字段。

产出：更新 `glossary.md`（补充 DDD 相关字段）

#### 7. 双向校验与差异报告

将以上分析结果与 wiki 中的代码事实对比：

- wiki 中的 `domains/` 划分 vs DDD 分析的上下文划分——边界是否一致？
- wiki 中的 `interfaces/` 关系类型 vs DDD 分析的映射关系——是否完整？
- wiki 中的模块归属 vs DDD 分析的上下文归属——有没有模块被错分？
- 代码中实际存在的依赖 vs DDD 期望的依赖——有没有不该存在的耦合？

对每个差异标注：
- 差异描述
- 严重程度（提示 / 警告 / 阻塞）
- 建议行动

产出：`ddd/gaps.md`（仅记录战略层发现的初始差异）

### 战略层完成标准

战略层做完后，必须能回答这个问题：

> 从现在的烟囱状态，到底有几个限界上下文？每个上下文的对外契约是什么？

回答不出来，就是没做完。向用户呈现总结时，显式检验这个标准。

### 战略层产出汇总

| 文件 | 内容 |
|------|------|
| `ddd/status.md` | 状态总览（每次操作后更新） |
| `ddd/decisions.md` | 先决决策（架构定位、范围、目标） |
| `ddd/panorama.md` | 领域全景图（能力盘点 + 子域分类 + 烟囱报告） |
| `ddd/contexts.md` | 限界上下文（定义 + 映射关系） |
| `ddd/gaps.md` | 初始差异报告（战略层发现） |
| `glossary.md`（增强） | 统一语言（DDD 字段） |

## 页面模板

### ddd/decisions.md

```yaml
---
type: ddd-decisions
created: {YYYY-MM-DD}
last_updated: {YYYY-MM-DD}
---
```

必含章节：梳理范围（repo 清单）、梳理目标、输出偏好、系统架构定位（仅横切系统）。

### ddd/panorama.md

```yaml
---
type: ddd-panorama
scope: [纳入梳理的 repo 列表]
created: {YYYY-MM-DD}
last_updated: {YYYY-MM-DD}
---
```

必含章节：

- **业务能力盘点**：表格列出所有业务能力（能力名 / 描述 / 暴露方式 / 所属子域 / 实现模块）
- **子域分类**：核心域 / 支撑域 / 通用域三类，每类列出包含的能力，标注分类依据
- **子域全景图**：ASCII art 或 Mermaid 图，展示子域之间的关系
- **烟囱报告**：每组烟囱一个子章节（烟囱名称 / 存在位置 / 差异分析 / 合并建议）

### ddd/contexts.md

```yaml
---
type: ddd-contexts
context_count: {N}
created: {YYYY-MM-DD}
last_updated: {YYYY-MM-DD}
---
```

**上下文定义**——每个上下文一个章节：

- **职责边界**：做什么、不做什么
- **对外契约**：暴露的接口（命令 + 查询）
- **核心数据**：该上下文"拥有"的实体
- **所属子域**：核心 / 支撑 / 通用
- **实现现状**：当前由哪些 repo/module 承载（链接到 wiki 页面）
- **边界判据**：为什么这样划分（数据边界 / 语言边界 / 团队边界 / 变更频率）

**映射关系**——合并在同一文件中：

- **映射全景图**：ASCII art 或 Mermaid 图
- **关系明细**：表格（上游 / 下游 / 关系类型 / 集成方式 / 数据一致性 / 备注）
- **关系类型统计**：各类型的数量分布

### ddd/gaps.md

```yaml
---
type: ddd-gaps
gap_count: {N}
created: {YYYY-MM-DD}
last_updated: {YYYY-MM-DD}
---
```

战略层初始差异报告。每个差异一行或一节：差异描述、类型（边界不一致 / 关系缺失 / 模块错分 / 非法耦合）、严重程度（提示 / 警告 / 阻塞）、相关页面、建议行动。
