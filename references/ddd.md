# 反向 DDD 梳理方法论

> 本文件是 `/lcw ddd` 系列命令的详细指引。执行 DDD 梳理前必须阅读。

## 核心理念

你做的是**反向 DDD**——从代码考古出发重建领域模型，不是从需求出发设计领域模型。

最大的陷阱是**被代码现状带跑**：把烟囱当领域，把实现细节当业务概念。对抗这个陷阱的方法是**双向校验**：

- **自下而上**：从 `__wiki__/` 中提取代码事实（模块、接口、数据结构、调用关系）
- **自上而下**：从业务能力出发推导领域应该是什么样的
- **差异分析**：两边对不上的地方才是真正需要关注的问题

wiki 中的内容遵循**三层视角模型**（详见 SCHEMA.md "视角约定"）：

- **现状理解层**（`__wiki__/` 根目录）：代码中实际呈现的结构。由 ingest/diff/lint 维护。
- **目标设计层**（`__wiki__/ddd/`）：领域模型应该是什么样。由 ddd 命令维护。
- **差距层**（`glossary.md` 的"差距"字段 + `ddd/gaps.md`）：两者的不一致 = 治理清单。

特殊的**双视角文档**：`glossary.md` 同时包含现状用法（As-Is）和领域定义（To-Be）。统一语言不单独建文件，直接增强 glossary.md 的条目。

## 命令结构

```
/lcw ddd                          # 交互式全流程
/lcw ddd strategic                # 战略层：领域识别
/lcw ddd tactical <context-name>  # 战术层：指定上下文的模型落地
/lcw ddd evolution                # 演进层：重构推进规划
/lcw ddd audit                    # 漂移检测：代码现实 vs DDD 模型
```

无参数时的默认行为：
1. 检查 `__wiki__/ddd/` 是否已有产出
   - 有 → 读取进度，从上次断点继续
   - 无 → 从先决判断开始
2. 战略层完成后，交互式询问：先做哪个上下文的战术层？
3. 战术层完成一个后提示：继续下一个还是进入演进规划？

**迭代节奏**：战略层粗做一遍 → 挑一个上下文做战术层 → 反过来修正战略层 → 再做下一个上下文。这个往返是正常的，不要等战略层"完美"再开始战术层。但统一语言词汇表必须在最早建立并持续维护。

## 第零步：先决判断

首次运行 `/lcw ddd` 时，交互式引导用户做以下决策，持久化到 `__wiki__/ddd/decisions.md`。已有决策文件时跳过此步（但用户可随时用 `/lcw ddd` 重新审视）。

### 1. 系统架构定位

如果系统属于横切类型（治理、安全、合规、监控等嵌入所有业务的系统），必须先定位架构模式：

| 模式 | 特征 | 适用场景 |
|------|------|---------|
| 集中式平台 | 核心能力集中暴露，业务通过接口/事件接入 | 能力标准化程度高、业务侧只需调用 |
| 分布式框架 | 提供 SDK 和扩展点，规则下沉到业务侧 | 业务侧差异大、需要深度定制 |
| 两层分离 | 规则与编排集中，执行点嵌入业务 | 大多数治理类系统的终态 |
| 不适用 | 系统不是横切类型 | 纯业务系统 |

这个判断不先做，后面的限界上下文划分会反复推翻。

### 2. 梳理范围与目标

- **范围**：哪些 repo 纳入本次梳理？从 `__wiki__/repos/` 中列出清单供用户选择
- **目标**：梳理要解决什么问题？
  - 理解现状（新人 onboarding / 团队对齐）
  - 规划重构（烟囱合并 / 架构升级）
  - 支撑新需求设计（在已有架构上长出新能力）
  - 持续治理（建立领域模型并保持与代码同步）

### 3. 输出格式偏好

- 词汇表语言：中文 / 英文 / 双语
- 图表格式：ASCII art / Mermaid / 纯文字描述

## 第一层：战略层

命令：`/lcw ddd strategic`

### 输入

从 `__wiki__/` 中读取：

| 页面类型 | 用途 |
|---------|------|
| `overview.md` | 系统全景、业务能力地图 |
| `domains/*.md` | 已识别的领域（代码事实层面） |
| `interfaces/*.md` | 跨系统接口、已有 DDD 关系类型标注 |
| `glossary.md` | 术语表 |
| `flows/*.md` | 端到端业务流程 |
| `modules/*.md` | 模块详情（按需深入） |
| `issues/*.md` | 已知问题（可能揭示领域边界问题） |

如果 wiki 尚未摄入或内容不足，提示用户先执行 `/lcw ingest`。

### 分析步骤

#### 1. 业务能力盘点

从 wiki 的 domains 和 modules 中提取系统对外暴露的所有业务能力：

- 读取每个 `domains/*.md` 的 `capabilities` 字段
- 读取每个 `modules/*.md` 的 Public API 章节
- 去重合并，按业务含义（而非代码结构）分组
- 标注每个能力：对外暴露 / 仅内部使用 / 未明确

产出：能力清单（写入 `ddd/panorama.md` 的"业务能力盘点"章节）

#### 2. 烟囱检测

找"实质上做同一件事"的多个实现：

- 在 wiki 中搜索相似的 API 签名、数据结构、业务逻辑描述
- 对比不同模块中名称相似或职责重叠的组件
- 重点关注：同一业务操作在不同入口（HTTP API / RPC / 定时任务 / 消息消费者）的多份实现
- 对每个烟囱记录：存在于哪些模块、差异点、统一的阻碍

产出：烟囱报告（`ddd/silos.md`）

#### 3. 子域分类

将能力分组为三类：

| 类型 | 判据 | 投入策略 |
|------|------|---------|
| 核心域 | 直接产生业务差异化价值的能力 | 重点投入、精心建模 |
| 支撑域 | 业务必需但不直接产生差异化的能力 | 标准化、够用即可 |
| 通用域 | 通用基础设施能力（无业务特殊性） | 尽量复用外部方案 |

分类时警惕两个常见错误：
- 把所有东西都标为"核心"——如果一切都是核心，就没有核心
- 把重要性等同于核心域——"重要"不等于"差异化"，数据库很重要但通常是通用域

这一步需要人工确认——呈现分类方案，说明判据，让用户调整。

产出：更新 `ddd/panorama.md` 的"子域分类"章节

#### 4. 限界上下文划分

基于能力分组 + 数据边界 + 团队边界，提出上下文划分方案：

- 一个限界上下文 = 一组内聚的业务能力 + 统一的领域语言 + 明确的边界
- 边界判据：
  - 数据边界：哪些实体只在这个上下文内被修改？
  - 语言边界：同一个词在不同地方含义不同？（从 glossary.md 的"多义"条目获取）
  - 团队边界：谁负责这部分代码？（从 activities/ 的作者数据获取）
  - 变更频率：一起变更的代码通常属于同一个上下文

对每个上下文写清楚：
- 职责边界（做什么、不做什么）
- 对外契约（暴露什么接口）
- 拥有的核心数据（哪些实体归这个上下文"拥有"）

产出：`ddd/contexts.md`

#### 5. 上下文映射

从 `interfaces/*.md` 中提取现有的跨上下文关系，补充遗漏的：

| 关系类型 | 含义 | 老系统常见度 |
|---------|------|------------|
| Partnership | 两个上下文紧密合作，同步演进 | 低 |
| Customer-Supplier | 上游提供，下游消费，上游主导 | 中 |
| Conformist | 下游完全迁就上游模型 | 中 |
| Anti-Corruption Layer | 下游用转换层隔离上游的脏模型 | 高 |
| Shared Kernel | 两个上下文共享一小部分模型 | 中 |
| Open Host Service | 上游提供标准化协议供多下游使用 | 中 |
| Separate Ways | 各自独立，不集成 | 高 |

老系统改造里"Separate Ways"和"ACL"用得最多——不要回避承认这一点。

产出：`ddd/context-map.md`

#### 6. 统一语言构建

直接增强 `glossary.md`（双视角文档），不单独生成词汇表文件。

以 `glossary.md` 现有条目为基础：

- 提取所有 status 为"不一致"和"多义"的术语
- 对每个核心名词，在条目中补充 DDD 字段：
  - **领域定义（To-Be）**：该术语在领域模型中的精确定义。如果需要拆分为多个概念，列出每个概念的名称和定义
  - **差距**：现状用法与领域定义之间的不一致
  - **决策**：消解差距的行动方向
  - **消解状态**：未开始 / 进行中 / 已消解
- 标注哪些不一致是"历史遗留但无害"，哪些是"必须修正否则会持续引发误解"

写入时**保留** glossary.md 中由 ingest/diff/lint 维护的"现状用法"字段。

产出：更新 `glossary.md`（补充 DDD 相关字段）

#### 7. 双向校验与差异报告

将以上分析结果与 wiki 中的代码事实对比：

- wiki 中的 `domains/` 划分 vs DDD 分析的上下文划分——边界是否一致？
- wiki 中的 `interfaces/` 关系类型 vs DDD 分析的映射关系——是否完整？
- wiki 中的模块归属 vs DDD 分析的上下文归属——有没有模块被错分？
- 代码中实际存在的依赖 vs DDD 期望的依赖——有没有不该存在的耦合？

对每个差异标注：
- 差异描述
- 严重程度（提示 / 警告 / 阻塞）
- 建议行动

产出：`ddd/gaps.md`

### 战略层完成标准

战略层做完后，必须能回答这个问题：

> 从现在的烟囱状态，到底有几个限界上下文？每个上下文的对外契约是什么？

回答不出来，就是没做完。向用户呈现总结时，显式检验这个标准。

## 第二层：战术层

命令：`/lcw ddd tactical <context-name>`

对指定的限界上下文做深度建模。`<context-name>` 必须是 `ddd/contexts.md` 中已定义的上下文名称。

不需要一次性做完所有上下文。先挑一两个核心上下文做透，验证方法可行，再铺开。

### 输入

- `ddd/contexts.md` — 该上下文的职责边界
- `glossary.md` — 统一语言（读取"领域定义"字段）
- 该上下文涉及的 `modules/*.md` — 模块详情
- 该上下文涉及的 `domains/*.md` — 领域页面
- 源代码 — 按需读取验证（wiki 描述可能过时，关键判断要回源码确认）

### 分析步骤

#### 1. 数据模型提取

从 wiki 的模块页面提取现有数据结构：

- `modules/*.md` 的 Public API 和 Internal logic 章节
- `domains/*.md` 的"核心实体与聚合"章节
- 关注 frontmatter 中的 `model_style` 字段

#### 2. 聚合根识别

聚合根 = 一致性边界的守护者。识别判据：

- 哪些实体的修改必须是原子的？（事务边界）
- 哪个实体"拥有"其他实体的生命周期？（创建和销毁权）
- 外部引用只能通过谁的 ID 来访问？（引用入口）

不要照搬现有的数据库表结构。一张"上帝表"可能对应多个聚合根，多张关联表可能只是一个聚合的不同视角。

对每个聚合根记录：
- 聚合根实体
- 聚合内实体（被聚合根管理生命周期的实体）
- 值对象（无独立身份的属性组合）
- 不变量约束（invariants，该聚合必须始终满足的业务规则）

#### 3. 贫血检测

如果 `model_style` 是 `anemic` 或 `procedural`：

- 找出所有对该实体数据进行操作的 Service / Manager / Handler / Processor
- 判断这些操作中哪些是实体的内聚职责（应该搬进实体）
- 哪些是跨聚合的协调逻辑（应该放在领域服务或通过领域事件解耦）
- 哪些纯粹是基础设施关注点（应该放在应用层或基础设施层）

标注每个操作的建议归属，但不强制要求立即重构——这是现状分析，不是重构计划。

#### 4. 领域事件提取

从 `flows/*.md` 和 `modules/*.md` 中提取：

- 状态变更（entity.Status 从 A 变为 B 时，谁需要知道？）
- 异步通知（消息队列、事件总线中的消息）
- 跨上下文的数据同步（一个上下文的变更触发另一个上下文的动作）

对每个领域事件记录：
- 事件名称（过去时态，如 `PolicyEvaluated`、`RuleSetUpdated`）
- 触发条件
- 生产者（哪个聚合根产出）
- 消费者（谁订阅）
- 载荷（携带哪些数据）
- 是否需要持久化（事件溯源 vs 即发即弃）

#### 5. 防腐层设计

对该上下文与外部系统的每个交互点：

- 现有接口是"干净"的还是"脏"的？（脏 = 暴露了对方的内部实现细节）
- 是否需要 ACL 隔离？如果需要，转换层应该做什么？
- ACL 的方向：是保护本上下文不被外部污染，还是保护外部不被本上下文的变更影响？

防腐层是老系统重构的核心手段——用 ACL 把"旧的脏接口"和"新的干净模型"隔开，新模型可以在不动老代码的前提下先长出来。

#### 6. 领域服务识别

跨聚合的逻辑放在领域服务里，但要警惕"杂物间"倾向：

- 领域服务应该只包含**需要协调多个聚合的业务逻辑**
- 如果一个领域服务只操作一个聚合 → 这个逻辑应该在聚合根里
- 如果一个领域服务变成了几十个方法的大类 → 需要拆分或重新审视聚合边界

### 产出物

| 文件 | 内容 |
|------|------|
| `ddd/tactical/{context-name}.md` | 聚合根清单 + 实体 + 值对象 + 不变量 + 贫血检测结果 |
| `ddd/tactical/{context-name}-events.md` | 领域事件目录 |
| `ddd/tactical/{context-name}-acl.md` | 防腐层设计 |
| `ddd/tactical/{context-name}-services.md` | 领域服务清单（含杂物间风险标注）|

### 战术层完成后

完成一个上下文的战术层后：
1. 回看 `ddd/contexts.md`——这个上下文的边界定义是否需要调整？如果是，更新战略层产出
2. 回看 `glossary.md`——是否发现了新的术语不一致？如果是，补充 DDD 字段（领域定义、差距、决策）
3. 提示用户：继续下一个上下文，还是进入演进规划？

## 第三层：演进层

命令：`/lcw ddd evolution`

将战略层和战术层的产出落到工程节奏上。前置条件：战略层已完成，且至少一个核心上下文的战术层已完成。

### 输入

- 所有 `ddd/` 目录下的产出
- `__wiki__/issues/*.md` — 已知问题
- `__wiki__/activities/*.md` — 变更频率数据（如果有）

### 分析步骤

#### 1. 现状成熟度评估

对每个限界上下文评估：

| 维度 | 来源 | 评分标准 |
|------|------|---------|
| 烟囱程度 | `ddd/silos.md` | 有多少重复实现？ |
| 耦合度 | `ddd/context-map.md` + `interfaces/*.md` | 跨上下文依赖有多紧？ |
| 模型健康度 | `ddd/tactical/*.md` | 贫血程度、聚合边界清晰度 |
| 变更频率 | `activities/*.md` | 过去 N 个月改动频率 |
| 问题密度 | `issues/*.md` | 该区域的问题数量和严重度 |

#### 2. 优先级排序

重构优先级 = 耦合度高 × 变更频率高 × 问题密度高 的交集。

不是按架构师偏好排序，而是按"不改的成本最高"排序。明确标注排序依据，让用户可以挑战和调整。

#### 3. 迁移路径规划

对每个需要重构的上下文：

- **目标态**：DDD 模型中这个上下文应该是什么样？
- **过渡态**：中间态什么样？承认"过渡架构会存在很长时间"比假装"半年就能切完"更务实
- **步骤**：
  1. 先建 ACL 隔离新旧
  2. 在 ACL 后面长出新模型
  3. 逐步把流量从旧路径切到新路径
  4. 旧代码退役
- **灰度方案**：怎么逐步切流量？
- **回滚预案**：出问题怎么回退？

#### 4. 质量基线

定义重构期间不能退化的指标：

- 功能正确性（核心业务流程是否正常）
- 性能指标（延迟、吞吐）
- 可观测性覆盖（日志、监控、告警是否完备）
- 特定于业务领域的指标（例如治理类系统的：卡控有效性、决策延迟、误判率）

没有基线，重构后只能口头说"应该没退化"。

### 产出物

| 文件 | 内容 |
|------|------|
| `ddd/evolution/roadmap.md` | 重构路线图：优先级排序 + 依据 + 预估节奏 |
| `ddd/evolution/transition.md` | 过渡架构描述：每个中间态的架构图 |
| `ddd/evolution/migration-{context}.md` | 逐上下文的迁移指南、灰度方案、回滚预案 |
| `ddd/evolution/baseline.md` | 质量基线定义与回归指标 |

## 漂移检测

命令：`/lcw ddd audit`

核心目的：把 DDD 从一次性梳理变成持续治理。

### 执行逻辑

1. 读取 `__wiki__/ddd/` 下所有产出（DDD 模型 = "应然态"）
2. 读取 `__wiki__/` 根目录的最新代码事实页面（代码现实 = "实然态"）
3. 对比分析：

| 检查项 | 对比方式 | 发现问题时的处理 |
|--------|---------|----------------|
| 上下文边界 | `ddd/contexts.md` vs `domains/*.md` | 有新模块不属于任何已定义的上下文？ |
| 映射关系 | `ddd/context-map.md` vs `interfaces/*.md` | 有新接口未在映射中？关系类型变了？ |
| 统一语言 | `glossary.md` 中的"领域定义"vs"现状用法" | 有新的不一致术语出现？已消解的差距是否又回退？ |
| 聚合边界 | `ddd/tactical/*.md` vs `modules/*.md` | 聚合内新增了不该有的依赖？ |
| 事件目录 | `ddd/tactical/*-events.md` vs `flows/*.md` | 有新的跨上下文通信未通过事件？ |

4. 产出差异报告，对每个差异标注：
   - **代码追上了模型**：代码变更使得实际结构更接近 DDD 模型 → 正面信号
   - **代码偏离了模型**：代码变更背离了 DDD 模型 → 需要关注
   - **模型需要更新**：业务本身变了，DDD 模型已过时 → 更新模型

5. 产出写入 `ddd/gaps.md`（覆盖更新）

### 推荐用法

每次执行 `/lcw ingest` 或 `/lcw diff` 后，跑一次 `/lcw ddd audit`。可以在 `/lcw diff` 或 `/lcw lint` 完成后自动提示用户。

## 与 wiki 的集成约定

### 读取（DDD → wiki）

| wiki 页面 | DDD 怎么用 |
|-----------|-----------|
| `domains/*.md` | 子域划分的原料，但 DDD 可能重新划分边界 |
| `interfaces/*.md` | 复用 relationship 标注，补充遗漏的映射关系 |
| `glossary.md` | 统一语言的起点 |
| `modules/*.md` 的 `model_style` | 贫血检测的直接输入 |
| `flows/*.md` | 领域事件提取的主要来源 |
| `activities/*.md` | 变更频率、团队边界的数据源 |
| `issues/*.md` | 问题密度评估 |

### 回写（DDD → wiki）

DDD 分析过程中发现的内容回写到 wiki：

- 新发现的问题 → 创建 `issues/*.md`
- 术语的领域定义和差距 → 更新 `glossary.md` 条目的 DDD 字段（领域定义、差距、决策、消解状态）
- 新发现的概念 → 创建 `concepts/*.md`
- 领域的 DDD 定位 → 更新 `domains/*.md` 的 `ddd_context`/`ddd_status` frontmatter 字段 + 底部"DDD 视角"章节

DDD 不修改 `modules/*.md`、`repos/*.md` 的主体内容——这些是代码事实，由 ingest/diff/lint 维护。对于 `domains/*.md`，DDD 只写入 frontmatter 的 DDD 字段和底部的"DDD 视角"章节，不修改主体内容。对于 `glossary.md`，DDD 只写入 DDD 字段，不修改"现状用法"。

## 常见陷阱

在执行过程中时刻警惕：

1. **把烟囱当领域**：三个团队各自做了一套"风控"，不代表有三个风控域——很可能只有一个核心风控域和两个应该被合并的烟囱
2. **把贫血模型当领域模型**：wiki 抽出来的数据结构如果直接拿来当领域模型，大概率是贫血的。必须把"行为"补回模型里
3. **回避难听的结论**：大量的"Separate Ways"和"ACL"在老系统里是常态，不要美化为"Partnership"
4. **子域分类一刀切**：团队容易把自己负责的所有东西都标为"核心域"。核心域的判据是"差异化价值"，不是"重要性"
5. **忽略团队边界**：康威定律是真实的。上下文划分必须考虑团队结构，否则会产生"组织上无法维护"的领域模型
6. **过早追求完美**：战略层的上下文划分永远不会在第一遍就完美。先做粗分，在战术层验证后再修正
7. **产出文档化但不被使用**：DDD 的最终产出不是文档，是团队共识。通过 audit 机制保持 DDD 模型与代码的持续对齐，让文档保持活性
