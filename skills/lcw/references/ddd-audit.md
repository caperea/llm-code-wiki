# 反向 DDD——漂移检测

> 前置依赖：先读 `references/ddd-core.md`。本文件覆盖漂移检测逻辑 + 页面模板。

## 漂移检测

命令：`/lcw ddd audit`

核心目的：把 DDD 从一次性梳理变成持续治理。

### 执行逻辑

1. 读取 `ddd/` 下所有产出（DDD 模型 = "应然态"）
2. 读取 wiki 根目录的最新代码事实页面（代码现实 = "实然态"）
3. 对比分析：

| 检查项 | 对比方式 | 发现问题时的处理 |
|--------|---------|----------------|
| 上下文边界 | `ddd/contexts.md` vs `domains/*.md` | 有新模块不属于任何已定义的上下文？ |
| 映射关系 | `ddd/contexts.md` 映射章节 vs `interfaces/*.md` | 有新接口未在映射中？关系类型变了？ |
| 统一语言 | `glossary.md` 中的"领域定义"vs"现状用法" | 有新的不一致术语出现？已消解的差距是否又回退？ |
| 聚合边界 | `ddd/tactical/*.md` 的"聚合根清单"章节 vs `modules/*.md` | 聚合内新增了不该有的依赖？ |
| 事件目录 | `ddd/tactical/*.md` 的"领域事件"章节 vs `flows/*.md` | 有新的跨上下文通信未通过事件？ |

4. 产出差异报告，对每个差异标注：
   - **代码追上了模型**：代码变更使得实际结构更接近 DDD 模型 → 正面信号
   - **代码偏离了模型**：代码变更背离了 DDD 模型 → 需要关注
   - **模型需要更新**：业务本身变了，DDD 模型已过时 → 更新模型

5. 产出写入 `ddd/audit-{YYYY-MM-DD}.md`（每次审计独立文件，不覆盖历史）

### 推荐用法

每次执行 `/lcw ingest` 或 `/lcw sync` 后，跑一次 `/lcw ddd audit`。可以在 `/lcw sync` 或 `/lcw lint` 完成后自动提示用户。

### audit 与 gaps 的关系

- `ddd/gaps.md`：战略层步骤 7 产出的**初始差异报告**，记录分析时发现的 As-Is 与 To-Be 之间的结构性差异。相对稳定，随战略层修订更新。
- `ddd/audit-*.md`：每次审计产出的**增量漂移报告**，记录代码变更后与 DDD 模型的偏差。随时间积累。

两者职责不同：gaps 是"一开始就知道的差距"，audit 是"后来漂出来的偏差"。

## 页面模板

### ddd/audit-{YYYY-MM-DD}.md

```yaml
---
type: ddd-audit
audit_date: {YYYY-MM-DD}
diff_count: {N}
---
```

增量漂移报告。每个差异标注类型：代码追上了模型（正面）/ 代码偏离了模型（关注）/ 模型需要更新（过时）。
