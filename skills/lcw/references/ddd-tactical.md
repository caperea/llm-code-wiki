# 反向 DDD——战术层

> 前置依赖：先读 `references/ddd-core.md`。本文件覆盖战术层分析步骤 + 页面模板。

## 第二层：战术层

命令：`/lcw ddd tactical <context-name>`

对指定的限界上下文做深度建模。`<context-name>` 必须是 `ddd/contexts.md` 中已定义的上下文名称。

不需要一次性做完所有上下文。先挑一两个核心上下文做透，验证方法可行，再铺开。

**按子域类型调整深度**：

| 子域类型 | 战术层深度 | 做什么 | 不做什么 |
|---------|----------|--------|---------|
| 核心域 | 全量（7 步） | 全部分析步骤 | — |
| 支撑域 | 精简 | 聚合根识别 + 领域事件 + 防腐层 | 贫血检测、领域服务详细分析 |
| 通用域 | 通常跳过 | 仅在耦合严重时做防腐层设计 | 完整战术建模 |

投入应该跟着业务价值走——对通用域做精细建模是浪费。

### 输入

- `ddd/contexts.md` — 该上下文的职责边界和映射关系
- `glossary.md` — 统一语言（读取"领域定义"字段）
- 该上下文涉及的 `modules/*.md` — 模块详情
- 该上下文涉及的 `domains/*.md` — 领域页面
- 该上下文涉及的 `flows/*.md` — 端到端业务流程（Event Storming 的 markdown 版本——每个 flow 已按 触发→命令→聚合→领域事件 结构组织，是提取领域事件和聚合根的主要来源）
- 源代码 — 按需读取验证（wiki 描述可能过时，关键判断要回源码确认）

### 分析步骤

#### 1. 数据模型提取

从 wiki 的模块页面提取现有数据结构：

- `modules/*.md` 的 Public API 和 Internal logic 章节
- `domains/*.md` 的"核心实体与聚合"章节
- 关注 frontmatter 中的 `model_style` 字段

#### 2. 聚合根识别

聚合根 = 一致性边界的守护者。识别判据：

- 哪些实体的修改必须是原子的？（事务边界）
- 哪个实体"拥有"其他实体的生命周期？（创建和销毁权）
- 外部引用只能通过谁的 ID 来访问？（引用入口）

不要照搬现有的数据库表结构。一张"上帝表"可能对应多个聚合根，多张关联表可能只是一个聚合的不同视角。

对每个聚合根记录：
- 聚合根实体
- 聚合内实体（被聚合根管理生命周期的实体）
- 值对象（无独立身份的属性组合）
- 不变量约束（invariants，该聚合必须始终满足的业务规则）

#### 3. 贫血检测

如果 `model_style` 是 `anemic` 或 `procedural`：

- 找出所有对该实体数据进行操作的 Service / Manager / Handler / Processor
- 判断这些操作中哪些是实体的内聚职责（应该搬进实体）
- 哪些是跨聚合的协调逻辑（应该放在领域服务或通过领域事件解耦）
- 哪些纯粹是基础设施关注点（应该放在应用层或基础设施层）

**贫血不一定是问题**。在 Go/Rust 等语言中 procedural 风格可能是刻意的技术选择。真正的问题信号是：同一组数据的操作散布在 5+ 个 Service 中且互相不知道对方的存在——这意味着业务规则没有内聚的守护者，修改一个规则需要同时改多处。如果逻辑集中在少数几个 Service 中且职责清晰，procedural 风格本身不构成问题。

标注每个操作的建议归属，但不强制要求立即重构——这是现状分析，不是重构计划。

#### 4. 领域事件提取

从 `flows/*.md` 和 `modules/*.md` 中提取（`flows/*.md` 已按 Event Storming 结构组织，是最高效的事件来源）：

- 状态变更（entity.Status 从 A 变为 B 时，谁需要知道？）
- 异步通知（消息队列、事件总线中的消息）
- 跨上下文的数据同步（一个上下文的变更触发另一个上下文的动作）

对每个领域事件记录：
- 事件名称（过去时态，如 `PolicyEvaluated`、`RuleSetUpdated`）
- 触发条件
- 生产者（哪个聚合根产出）
- 消费者（谁订阅）
- 载荷（携带哪些数据）
- 是否需要持久化（事件溯源 vs 即发即弃）

#### 5. 防腐层设计

对该上下文与外部系统的每个交互点：

- 现有接口是"干净"的还是"脏"的？（脏 = 暴露了对方的内部实现细节）
- 是否需要 ACL 隔离？如果需要，转换层应该做什么？
- ACL 的方向：是保护本上下文不被外部污染，还是保护外部不被本上下文的变更影响？

防腐层是老系统重构的核心手段——用 ACL 把"旧的脏接口"和"新的干净模型"隔开，新模型可以在不动老代码的前提下先长出来。

#### 6. 领域服务识别

跨聚合的逻辑放在领域服务里，但要警惕"杂物间"倾向：

- 领域服务应该只包含**需要协调多个聚合的业务逻辑**
- 如果一个领域服务只操作一个聚合 → 这个逻辑应该在聚合根里
- 如果一个领域服务变成了几十个方法的大类 → 需要拆分或重新审视聚合边界

#### 7. 跨聚合协调模式

当一个业务流程跨越多个聚合根时，光有领域事件不够——需要明确的协调模式。遗留系统中常见的情况是一个大事务覆盖多个聚合，重构时必须选择协调策略：

| 模式 | 适用场景 | 一致性 | 复杂度 |
|------|---------|--------|--------|
| Saga（编排式） | 各聚合通过事件链自行推进 | 最终一致 | 低（但调试难） |
| Process Manager（协调式） | 中心协调器控制流程步骤 | 最终一致 | 中 |
| 大事务（现状） | 所有操作在同一事务中 | 强一致 | 低（但扩展差） |

对每个跨聚合的业务流程：
- 当前是用什么模式协调的？（通常是大事务或隐式的调用链）
- 目标态建议用哪种模式？
- 补偿逻辑（回滚/重试）现在有没有？缺了哪些？

不要默认推荐 Saga——如果当前的大事务在性能和规模上没有问题，记录现状即可，在演进层再规划迁移。

### 产出物

每个上下文一个文件：`ddd/tactical/{context-name}.md`，用章节组织所有战术层内容。

| 章节 | 内容 |
|------|------|
| 聚合根清单 | 聚合根 + 聚合内实体 + 值对象 + 不变量 + 贫血检测结果 |
| 领域事件 | 事件目录（名称、触发条件、生产者、消费者、载荷、持久化需求） |
| 防腐层 | 每个外部交互点的 ACL 设计（方向、转换规则、当前状态） |
| 领域服务 | 服务清单（职责、协调的聚合、杂物间风险标注） |
| 跨聚合协调 | 跨聚合业务流程的协调模式（Saga/Process Manager/大事务）及补偿逻辑 |

单文件的好处：一个上下文的所有战术模型在一处阅读和维护，不用在多个文件间跳转。当某个章节特别长（如聚合根超过 5 个、事件超过 20 个）时，可以拆分为独立文件并在主文件中用链接引用。

### 战术层完成后

完成一个上下文的战术层后：
1. 回看 `ddd/contexts.md`——这个上下文的边界定义是否需要调整？如果是，更新战略层产出
2. 回看 `glossary.md`——是否发现了新的术语不一致？如果是，补充 DDD 字段（领域定义、差距、决策）
3. 更新 `ddd/status.md` 进度表
4. 提示用户：继续下一个上下文，还是进入演进规划？

## 页面模板

### ddd/tactical/{context-name}.md

```yaml
---
type: ddd-tactical
context: {context-name}
aggregate_count: {N}
event_count: {N}
acl_count: {N}
service_count: {N}
created: {YYYY-MM-DD}
last_updated: {YYYY-MM-DD}
---
```

必含章节：

- **上下文概述**：一段话描述该上下文的核心职责（链接到 `[[ddd/contexts]]`）
- **聚合根清单**：每个聚合根一个子章节（聚合根实体、聚合内实体、值对象、不变量约束、代码现状）
- **贫血检测结果**：被错放在 Service/Manager 中的实体行为清单，标注建议归属
- **领域事件**：表格（事件名称 / 触发条件 / 生产者聚合 / 消费者 / 载荷摘要 / 持久化需求 / 当前实现方式）
- **防腐层**：每个 ACL 一个子章节（隔离对象、方向、转换规则、当前状态）
- **领域服务**：每个服务一行或一节（服务名称、职责、协调的聚合、杂物间风险、当前实现）
- **跨聚合协调**（核心域必含，支撑域按需）：每个跨聚合流程一节（业务流程名称、涉及聚合根、当前协调模式、目标协调模式、补偿逻辑现状）
