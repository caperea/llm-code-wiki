# /lcw file <name>

把查询中结晶出的洞察归档。归档内容按来源分流到不同位置。

**关键区分**：file 处理三种内容，去向不同：

## 一、代码事实的结晶 → wiki

查询或对话过程中，从代码中提炼出了值得独立记录的高阶理解（如跨模块的隐藏耦合、重复模式、未文档化的业务规则）。这些信息**来源于代码**，只是需要人提问才被结晶出来。

**去向**：融入 wiki 已有页面——
- 跨模块关系 → 补充到相关 `modules/{repo}/` 和 `domains/` 页面
- 业务规则 → 补充到 `domains/` 页面
- 接口发现 → 补充到 `interfaces/` 页面
- 新发现的问题 → 创建 `issues/` 页面
- 新发现的业务流程 → 创建 `flows/` 页面

如果内容确实不适合融入任何已有页面，说明可能发现了新的领域或模块——应该创建对应的 `domains/` 或 `modules/` 页面。

## 二、查询记录 → .inputs/queries/

查询本身（提问角度、思考路径、分析方法）作为 source 存入 `.inputs/queries/`。它是构建知识的原料，不是知识本身。

## 三、人类提供的非代码信息 → .inputs/notes/

用户在对话中提供了代码中不存在的信息——业务背景、历史决策原因、团队约定、口头知识等。

**去向**：存入 `.inputs/notes/`，带时间戳和来源标注。不直接写入 wiki 页面——后续 query/ingest/sync 过程中如果能用代码验证，再融入 wiki。

## 写入后

1. wiki 部分：添加 `[[wikilink]]`、反向链接、检查词汇表、更新 `index.md`
2. 所有写入：写 `log.md`
